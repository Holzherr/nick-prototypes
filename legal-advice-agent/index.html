Payment model
Stripe Connect · authorize → capture → release
1 · Authorize on approval
When the client approves the engagement (single approval step — terms + scope + payment), we authorize the fee on their card via Stripe. No charge yet — it's a hold.
2 · Capture on engagement
When the matched lawyer is engaged, we capture the hold. Funds sit in GitLaw's marketplace escrow.
3 · Release on acceptance
Client clicks "Accept the review" → Stripe pays out to the lawyer's connected account. Receipt + invoice filed in the repo's audit trail.
Cancel / dispute
If client cancels mid-review, lawyer submits a partial invoice; client pays partial fee and the rest of the hold is released. Disputes escalate to GitLaw support for arbitration.
Marketplace fee (internal note): GitLaw takes a marketplace fee from the lawyer's payout, in line with Rocket Lawyer / LegalZoom / Upwork practice. Not disclosed to the client.
Spec — Lawyer Loop-In (v0.4)
High-level intent · derived from eaglelaw-spec/10-feature-spec/20-ai-assistance/60-lawyer-loop-in.md
Problem

Users trust AI for speed but not for high-stakes legal output. They need fast, affordable human verification embedded in the same workflow.

Flow
  1. User picks a tier and asks the agent to find a lawyer.
  2. Agent reaches out to the panel (matter type only — no doc, no memory yet).
  3. Bids come back; agent ranks and surfaces the top match.
  4. User reviews the named lawyer + price, then accepts terms + scope + payment in one approval step.
  5. Lawyer is engaged, escrow capture happens, full context bundle delivered.
  6. Lawyer reviews in-doc; submits a Change Request + separate memo.
  7. User clicks Accept the review → escrow releases.
  8. User rates the engagement (optional) and decides whether to take the lawyer out of the repo or keep them in until signature.
Principles
  • Agent is orchestrator, not advisor. Avoids UPL risk in the US.
  • User sees the lawyer before paying. Match → name → consent → capture.
  • Single approval step. Terms + scope + payment authorization collapse into one click.
  • Granular consent on context. Per-scope toggles for doc, Profile, Playbook, contact files.
  • Acceptance triggers release. Escrow only releases when the user explicitly accepts the review.
  • User controls lawyer access. Take them out, or keep them in — user's call, not the lawyer's.
  • Feedback as native primitives. Suggestions + Comments + a separate written memo.
  • Marketplace fee invisible to client. Standard across legal & freelance marketplaces.
Pricing (prototype values)
  • Junior: £100 base + £0.08/word — ~£250, ~3h.
  • Senior: £200 base + £0.135/word — ~£450, ~1.5h, redline + memo.
  • Expert: £400 base + £0.22/word — ~£800, ~24h, partner sign-off.
Open forks
  1. Match vs choice. Agent auto-picks (current) vs surface 2-3 bids and let user choose.
  2. Repo model. Lawyer as Party with special posture vs sidecar service signing Change Requests as the firm.
  3. Privilege. Is the user the lawyer's direct client, or is GitLaw an intermediary?
  4. Revisions path. If user wants the lawyer to redo something before accepting, does that consume the same fee or is it a top-up?
  5. Memory-share granularity. Per-scope toggles (current) or a single "share my context" switch?
