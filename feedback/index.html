P3
extraction ↗ live on Stg-chat

Memory — extracted from chat

GitLaw extracts and remembers context across chat sessions. Have a real-ish session, then run /print-memory to see what it stored.

Key thing to test: memory should only store things uniquely useful for later turns. Not playbook content (already loaded), not generic chitchat, not restatements of what's in the chat. Fluff dilutes the signal and degrades the agent. Flag anything that doesn't pass: "would this uniquely help a future turn?"

What to look for

Accuracy

  1. Anything factually wrong — date, party, jurisdiction, clause text the agent got incorrect.
  2. Anything legally wrong — misclassified contract type, clause summarised in a way you'd push back on, wrong inference about precedent or governing law.
  3. Anything missing — facts a lawyer would have surfaced from the same conversation.
  4. Anything that shouldn't be stored — privileged content, off-topic chat treated as legal fact, advice-shaped statements rather than recall.

Materiality — equally important

  1. Flag stored items that don't pass "would this uniquely help a future turn?" — be aggressive.
  2. Anything stored that's already in a playbook? That's noise.

UX (welcome too)

  1. Does the agent reference memory naturally in later turns, or does it feel disconnected?
  2. Does memory shortcut work, or do you find yourself re-stating things?
  3. Is /print-memory readable for a non-engineer?

What to look for

Accuracy

  1. Categorisation — pick 2-3 contracts. Does the prototype's category match yours? Where does it get it wrong?
  2. Parties — counterparties, internal entities, signatories correctly identified?
  3. Dates — execution, effective, term, expiry. Anything off?
  4. Risk signals — does it surface non-standard clauses, or miss them?
  5. Missing fields — what would a lawyer NEED at-a-glance that isn't shown? Governing law, term, value caps, indemnity scope, etc.

UX (welcome too)

  1. First-week-as-GC — imagine you just joined a 5-year-old SaaS company. What would you click first? Which lens (categories vs parties vs status) is most useful?
  2. Compared to your current setup — DMS search, Outlook, folders? What's the realistic ceiling on adoption?

What to look for
  1. Trust gradient — for which changes would you accept immediately vs require manual review? Where's the line?
  2. Suggestion explanations — do the rationales help or clutter? When would you want more / less?
  3. Counterparty-facing output — after accepting, do the resulting track-changes look like yours, or do they give away the AI origin?
  4. Comments alongside redlines — useful to preserve AI reasoning as a comment? Or noise?
  5. What's missing — you do this all day in Word. What's the obvious gap?
  6. Mistake recovery — when the AI gets one wrong, is rejection smooth enough that you'd keep using it?

What to look for

P1 — BYO

  1. Lawyer onboarding — clear what you're being asked to do? Where would you hesitate?
  2. Scope — review only? Take over? Sign off? Where does it get fuzzy?
  3. Liability — where does it sit? Anything missing before you'd accept?
  4. Pricing — what should the user pay? What should the lawyer get?

P2 — AI agent

  1. Trust — where does it feel reliable vs risky? Specific examples are gold.
  2. UPL boundary — framing clear enough? Where does it cross a line?
  3. Escalation — when the agent offers to escalate, does it pass enough context?
  4. Substitution — would clients use this instead of calling you for work you want? Or only for noise you'd be glad to lose?

P1 vs P2

  1. If you had to pick one to ship in the next 6 months — which and why?
  2. What would make you trust P2 enough to default away from P1?
